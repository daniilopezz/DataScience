=========================
1. OBJETIVO DEL PROYECTO
=========================

El proyecto consiste en desarrollar una aplicación en Python conectada a una base de datos PostgreSQL que permita:

1. realizar login de usuarios que funcione desde la terminal, algo sencillo en esta parte
2. registrar sus accesos
3. registrar sus actividades dentro de la aplicación
4. analizar si una acción concreta parece normal o anómala
5. analizar si la sesión completa parece normal o anómala
6. detectar comportamientos sospechosos como:
   - uso por una persona distinta al usuario legítimo
   - intentos de robo de información
   - comportamiento automatizado tipo bot
   - ritmos de uso poco humanos
   - acceso a elementos raros o no habituales
   - secuencias de acciones poco naturales

La idea importante es que la detección de anomalías NO debe basarse principalmente en reglas hardcodeadas tipo if/else, sino en machine learning.

Las reglas de negocio pueden existir como apoyo documental o como base para generar datos sintéticos, pero la decisión principal de anomalía debe venir del modelo.

=========================
2. IDEA FUNCIONAL
=========================

La aplicación simula un sistema donde un usuario:

- hace login
- elige un elemento
- realiza una acción sobre ese elemento

Ejemplo de flujo:

- Login del usuario
- Menú de elementos:
  - FVG
  - AMCO
  - VETTING
  - RHODENSE
  - PAPARDO
  - PULEJO
- Menú de acciones:
  - Visualize
  - Create
  - Edit
  - Delete
  - Copy
  - Share

Cada acción debe:
- registrarse en base de datos
- pasar por un modelo de anomalía de actividad
- generar un “coste” o anomaly_probability
- acumularse en la sesión
- contribuir a una evaluación global de sesión

La sesión completa también debe ser evaluada por un segundo modelo de machine learning.

=========================
3. COMPORTAMIENTO QUE QUEREMOS DETECTAR
=========================

Queremos que el sistema aprenda hábitos distintos por usuario.

Ejemplo:
- Matteo hace normalmente 4 acciones por sesión
- Diego hace 12 o 15 por sesion
- Emilio puede hacer otro patrón

Por tanto:
- NO se puede poner una regla fija del tipo “si supera X acciones, es anomalía”
- el patrón normal depende del usuario
- el modelo debe aprender por usuario:
  - cuántas acciones suele hacer
  - a qué ritmo
  - a qué elementos entra
  - qué acciones realiza
  - cuánto tarda entre una acción y otra
  - qué secuencias parecen humanas
  - qué combinaciones parecen raras

También queremos que pueda detectar sesiones tipo:
- muchas acciones en muy poco tiempo
- repeticiones excesivas
- acciones raras para ese usuario
- cambios bruscos en el patrón habitual

9. QUÉ QUIERO A NIVEL MACHINE LEARNING
=========================

Quiero dos niveles de modelo:

A) MODELO DE ACTIVIDAD
Debe evaluar una acción individual.

Debe poder aprender cosas como:
- qué elemento es raro para ese usuario
- qué acción es rara para ese usuario
- qué entidad es rara para ese usuario
- si el timing entre acciones parece raro
- si el momento de la sesión en que ocurre la acción es raro

Features posibles:
- user_id
- element_id
- entity_id
- action_id
- hour
- minute
- day_of_week
- seconds_since_prev_action
- seconds_since_session_start
- rareza/frecuencia histórica de la combinación elemento-entidad-acción

Salida esperada:
- prediction
- anomaly_probability / cost

B) MODELO DE SESIÓN
Debe evaluar la sesión completa o parcial.

Debe aprender por usuario:
- cuántas acciones suele hacer
- cuánto dura la sesión
- cuánto tarda entre acciones
- si repite mucho lo mismo
- si la secuencia parece poco humana
- si el comportamiento acumulado se desvía del patrón habitual

Features posibles:
- action_count
- distinct_elements
- distinct_actions
- session_duration_min
- start_hour
- day_of_week
- actions_per_minute
- avg_seconds_between_actions
- min_seconds_between_actions
- max_seconds_between_actions
- cumulative_cost
- avg_cost
- max_cost
- repeated_action_ratio
- repeated_element_ratio

Muy importante:
quiero que este modelo tenga en cuenta que cada usuario puede tener un patrón de sesión distinto.


Si es posible que el modelo este todo en uno mismo para que a la hora de exportarlo solo sea uno en vez de dos archivos
